1. Component map — who calls who

Six hops, six numbers, in chronological order for one ticket. The detailed step-by-step is in section 2.

Chronological order for one ticket: ① poller ticks, queries Jira directly · ② a matching ticket is dispatched to langgraph-agent · ③ langgraph-agent calls back into jira-agent's MCP server to read the ticket (twice — get_ticket, then again via get_reporter) · ④ the ticket text goes to vLLM · ⑤ vLLM returns the parsed verdict (spam/valid, scope, comment) · ⑥ route writes the decision back to Jira via the same MCP server. ③ and ⑥ both terminate at jira-agent's MCP layer, which then executes the actual Jira REST call using the same jira.Client that edge ① uses for polling.

Color = who initiates the call: Go/jira-agent for ① and ②, Python/langgraph-agent for ③, ④ and ⑥, vLLM for the response leg ⑤. Neither service ever talks to Jira or vLLM through the other — jira-agent never calls vLLM, and langgraph-agent never calls Jira directly.

4. Why two services, not one

jira-agent holds the keys; langgraph-agent holds the judgment.

jira-agent (Go)

integration / infrastructure layer
  • Only thing holding JIRA_API_TOKEN
  • Owns polling schedule + JQL + label-based dedup
  • Exposes Jira as MCP tools over /mcp/jira
  • Never judges a ticket — just notices & later executes

langgraph-agent (Python)

decision / orchestration layer
  • Runs the LLM, makes every judgment call
  • Owns the LangGraph state machine
  • Zero Jira credentials of its own
  • Purely reactive — no ticker, only responds to /triage
