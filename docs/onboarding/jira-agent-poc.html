Architecture

Why Two Services? Separation of Concerns

Go for infrastructure reliability, Python for AI flexibility. Yes, it's two languages — here's why that trade-off made sense.

Go Monolith (jira-agent)

Infrastructure-critical code that must never crash:

  • Jira polling (every 30s)
  • MCP tool servers (Jira + K8s + Jenkins)
  • HITL approval watcher
  • Jira ticket transition handlers
  • Multi-cluster K8s support (ClusterManager)

Why Go?

Rock-solid concurrency, minimal dependencies, single binary deployment. When Jira polling crashes, engineers don't get notified — can't risk that.

Python Service (langgraph-agent)

Experimental AI layer that needs rapid iteration:

  • LangGraph orchestration
  • 5 specialized AI agents
  • Checkpoint persistence (K8s ConfigMaps)
  • Remediation lock service
  • FastAPI HTTP server

Why Python?

LangChain/LangGraph require it. OpenAI-compatible vLLM API, Pydantic structured output, rich AI ecosystem. Needs frequent prompt/agent changes.

Could this be one service? Yes. But mixing infrastructure code with AI felt a bit risky and out of place. This lets us restart the AI layer without touching the polling system. They communicate via MCP over HTTP/SSE — clean separation. A self-hosted vLLM instance serves Llama 3.1 70B on Habana Gaudi accelerators — no external API dependencies.

How It Works

Why Multi-Agent? Why Not One Big Script?

The LangGraph supervisor pattern orchestrates specialized agents. Each has a single responsibility — easier to test, trace, and debug.

The workflow loop

READ-ONLY
JiraAgent
→
Extract context, search history, find Jenkins URLs
READ-ONLY
JenkinsInvestigator
→
Build info, console log, failure type
READ-ONLY
K8sInvestigator
→
Multi-cluster state, logs, events
NO K8S ACCESS
Diagnostician
→
Root cause + remediation plan
HUMAN APPROVAL
Pause for /approve
→
Engineer reviews in Jira
WRITE ACCESS
Executor
→
Apply fix + verify
↻ If verification fails: retry with new diagnosis (max 2 attempts)

Why this pattern?

I tried a single-pass approach first

One LLM call to "diagnose and fix." It was brittle — when it failed, I couldn't tell which step broke. Each agent handles its relevant context, with steps completing fully before others start. The supervisor pattern enforces this.

Smart fallback paths

Not every ticket has K8s resources. The workflow detects this and routes to a synthesis node that diagnoses from Jenkins data + ticket history alone. Missing namespace? Posts an error back to Jira immediately.

Safety boundaries

Separating investigation (read-only) from remediation (write) means 3 agents literally cannot modify the cluster. Type system + transport layer enforce this.

Single responsibility

Each agent has one job. Easier to test in isolation. Easier to trace in LangSmith. Easier to replace one agent without breaking others.

Retry loop mimics humans

Humans debug by trying, checking, trying again with new info. The workflow does the same: diagnose → fix → verify → re-diagnose if needed.

Integration & Safety

Plugs Into Existing Infrastructure

How it integrates

  • Jira — polls for new tickets, auto-transitions status, posts results
  • Kubernetes — multi-cluster support (hldc02, hldc03), reads state, applies fixes
  • Jenkins — fetches build info, console logs, classifies failures
  • vLLM on Gaudi — self-hosted Llama 3.1 70B, no external API dependency
  • MCP Protocol — standardized tool interface (3 MCP servers: Jira, K8s, Jenkins)
  • LangSmith — full observability of every AI decision

Two-service architecture

Go monolith — Jira polling, MCP tool servers, HITL watcher

Python agent — LangGraph orchestration, AI agents, FastAPI

Built-in safety guardrails

🔒
Human-in-the-Loop Risk-based approval: high-risk actions require explicit Jira comment. Timeout tracking built in.
🛡
Read-Only Investigation 4 of 5 agents cannot modify the cluster. Type system enforces this.
🔓
Distributed Locking K8s ConfigMap-backed locks prevent concurrent remediation of the same resource
📈
Risk Classification Low-confidence findings skip remediation, post recommendations only
💾
Checkpoint Persistence Workflow state saved to K8s ConfigMaps. Survives restarts, enables HITL resume.
⚡
Rate Limiting 10 investigations/min, 60 HITL actions/min. Prevents abuse and runaway loops.

In Action

Real-World Example

Jira ticket: "order-service pods crashing in production after Jenkins deploy, users reporting 502 errors"

1. Jira Agent 30s

Reads ticket, extracts order-service in production namespace. Finds 3 similar past tickets. Extracts Jenkins build URL.

2. Jenkins Investigator 20s

Fetches build #456 info + console log. LLM classifies: infrastructure failure. Health check timeout during deploy.

3. K8s Investigator 60s

Finds 3 pods in CrashLoopBackOff. Logs show OutOfMemoryError. Events: OOMKilled x12. Memory limit: 256Mi.

4. Diagnostician 45s

Root cause: memory limit too low, confirmed by Jenkins deploy data. Historical match: 100% at 512Mi. Confidence: HIGH.

5. Human Approval ~2 min

Posts to Jira: "Propose increasing memory to 512Mi. Historical success: 100%."

Engineer comments: "approve"

6. Remediation 30s

Acquires lock. Sets memory to 512Mi. Waits for rollout. All 3 pods running and stable.

7. Result

Posts full investigation report to Jira with root cause, action taken, and verification. Labels ticket ai-agent-investigated.

Total time: ~6 minutes — Engineer effort: 30 seconds
